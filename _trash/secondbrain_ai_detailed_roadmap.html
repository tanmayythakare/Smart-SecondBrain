SecondBrain AI integration roadmap — 4 phases with detailed backend, frontend, API, DB, and checklist info for each

Phase 1 — MVP: Basic AI Chat

Build first. Get the full pipeline working end-to-end before adding memory.

Goal: user types a message → backend detects intent → Ollama is called → AI response returns with clickable task/note links → frontend renders them with routing to /tasks/:id or /notes/:id.

Backend New files

ai/AiChatController.java — POST /api/ai/chat, takes ChatRequestDto, returns ChatResponseDto, secured via existing JWT interceptor
ai/AiChatService.java — main orchestrator: calls intent detector, builds context, calls Ollama, validates returned IDs against DB, returns response
ai/OllamaService.java — thin HTTP client wrapping POST localhost:11434/api/generate with { model, prompt, stream:false }; must have try/catch returning graceful error on failure
ai/ContextBuilderService.java — fetches pending/overdue tasks (id, title, due_date) + last 5 note titles (id, title) from DB; assembles context string; hard cap 2000 tokens
ai/PromptBuilder.java — static utility, fills prompt template with context + user message; instructs AI to respond in JSON with message + items[]

Backend DTOs

dto/ai/ChatRequestDto.java
String message
dto/ai/ChatResponseDto.java
String message
List<LinkedItemDto> items
dto/ai/LinkedItemDto.java
Long id
String type — "task" or "note"
String title
LocalDate dueDate
Prompt template instructs AI to respond ONLY as JSON. Example:
{ "message":"…", "items":[{"id":12,"type":"task",…}] }

Frontend New feature module

features/ai-chat/ai-chat.component.ts — signal-based state: messages = signal<Message[]>([]), loading signal, sendMessage() method that appends user msg, calls service, appends AI response
features/ai-chat/ai-chat.service.ts — sendMessage(msg): Observable<ChatResponse> → POST /api/ai/chat; JWT interceptor covers this automatically
features/ai-chat/message-bubble/ — standalone component; user msgs right-aligned, AI msgs left-aligned; renders items[] as clickable pills below the message text
Clickable items use Angular Router: routerLink="/tasks/{{item.id}}" or "/notes/{{item.id}}" based on item.type
Loading state: show "AI is thinking..." indicator while request is in flight; Ollama cold starts can be slow

Frontend Routing + nav

Add to app routing: { path: 'ai', component: AiChatComponent, canActivate: [AuthGuard] }
Update navbar component to include "AI" tab with routerLink="/ai" and routerLinkActive="active"
API POST /api/ai/chat
Request: { message: "…" }
Response: { message: "…", items: […] }
Auth: JWT Bearer (auto via interceptor)
Risk AI may hallucinate IDs — validate every item ID in AiChatService against the DB before returning ChatResponseDto. Never trust raw Ollama output.

application.properties: ollama.base-url=http://localhost:11434

Phase 2 — Memory System + Smart Routing

Only start after Phase 1 is working end-to-end. Adds 3 DB tables, intent-based Ollama bypass, and compressed memory injection into context.

Goal: short, direct queries ("show pending tasks") skip Ollama entirely and return instantly. AI-required queries now include recent chat history and compressed past item references in context. Full memory pipeline operational.

Backend IntentDetectorService

New file: ai/IntentDetectorService.java — returns enum DIRECT_QUERY or AI_REQUIRED
DIRECT_QUERY keywords: "pending", "overdue", "today", "list my tasks", "show notes", "how many tasks", "what tasks" — keyword match on lowercased message
If DIRECT_QUERY: AiChatService queries DB directly, formats as ChatResponseDto, returns — no Ollama call, no latency
If AI_REQUIRED: full pipeline from Phase 1 runs, plus memory now injected into context
Expand keyword list incrementally as you discover common direct queries in production

Backend ContextBuilderService updates

Existing: pending/overdue tasks + last 5 note titles
Add: fetch last 10 messages from ai_chat_messages for this user (role + content), inject as [RECENT CHAT] block
Add: fetch compressed memory from ai_memory_compressed (id, title, due_date only), inject as [COMPRESSED MEMORY] block
Add: fetch session summaries from ai_chat_summaries for this user (last 3 summaries), inject as [PAST SESSIONS] block
Critical Enforce 2000 token hard limit — if total context exceeds it, drop compressed memory first, then past sessions, never drop the live task/note list

DB ai_chat_messages

Flyway: V{n}__create_ai_chat_messages.sql
id BIGSERIAL PRIMARY KEY
user_id BIGINT REFERENCES users(id)
session_id UUID NOT NULL
role VARCHAR(10) — 'user' or 'assistant'
content TEXT NOT NULL
created_at TIMESTAMP DEFAULT NOW()
Retention: 7 days full detail, then delete raw rows

DB ai_memory_compressed + ai_chat_summaries

ai_memory_compressed: id, user_id, item_type ('task'|'note'), item_id, title, due_date — only metadata, never full content
Populated: every time AI references a task or note in a response, record it here. Deleted: when the original item is deleted.
Retention: indefinite — this is how AI knows about old items without loading their content
ai_chat_summaries: id, user_id, session_id, summary TEXT, created_at
Trigger: session has >20 messages OR last message >24h ago → summarize and save, then delete raw messages
Retention: 30 days

What is NEVER stored in AI tables

Full note content — AI never receives or stores it. Notes are referenced by id + title only.
Full task content beyond id + title + due_date in compressed memory
User profile data, passwords, email addresses, any PII beyond what's in tasks/notes
Full chat history beyond 7 days — after that only summaries remain

Phase 3 — Note Assistant (in-note AI panel)

Requires Phase 1 + 2 stable. Adds an AI panel directly inside the note detail page with three actions.

Goal: user opens any note → sees an AI panel on the right → clicks Summarize, Explain, or Extract tasks → AI responds with note-specific output. Extracted tasks can be accepted with one click to create real tasks.

Backend New endpoint

New file: ai/NoteAiController.java → POST /api/ai/note-assist
Request body: { noteId: Long, action: String } where action is one of: "summarize", "explain", "extract_tasks"
New file: ai/NoteAiService.java — fetches note content from NoteRepository.findById(noteId), validates user owns the note (security), builds action-specific prompt
Summarize prompt: "Summarize this note in 2-3 sentences: {content}"
Explain prompt: "Explain the key concepts in this note clearly: {content}"
Extract tasks prompt: "Extract a list of actionable tasks from this note. Return as JSON: { tasks: [{title, suggestedDueDate}] }. Content: {content}"
Security Always verify note.userId == authenticatedUserId before fetching content — never allow cross-user note access

Frontend Note detail AI panel

Modify: features/notes/note-detail/note-detail.component.ts — add AI panel as a collapsible side section or bottom drawer
New: features/notes/note-detail/note-ai-panel/note-ai-panel.component.ts — takes @Input() noteId, manages its own loading/result state
Three action buttons: Summarize, Explain, Extract tasks — each calls POST /api/ai/note-assist with the relevant action string
For Extract tasks: render each suggestion as a card with title + suggested due date + "Add as task" button
"Add as task" button calls existing POST /api/tasks endpoint — reuses existing task creation, no new endpoint needed
Panel loading state: per-action spinner so the user knows which action is processing

API POST /api/ai/note-assist

Request: { "noteId": 42, "action": "extract_tasks" }
Response (summarize/explain): { "message": "This note covers…", "items": [] }
Response (extract_tasks): { "message": "Found 3 tasks", "items": [{"id":null,"type":"suggestion","title":"…","dueDate":"…"}] }
Note: extracted task suggestions use id: null and type: "suggestion" — they don't exist in DB yet, only created when user clicks "Add as task"

Phase 4 — Advanced / Optional

Only touch these after Phase 1–3 are stable and you have a concrete problem to solve. Don't build speculatively.

Three independent tracks — build whichever one your actual usage reveals as the bottleneck. They don't depend on each other.

Track A — Semantic search

Problem it solves: user has 500+ notes and "find the note about X" doesn't work with title-only matching
Approach: enable pgvector extension in PostgreSQL, add embedding vector(1536) column to notes table
On note save: call embedding API (Ollama's /api/embeddings endpoint or OpenAI), store vector
New endpoint: POST /api/ai/search — embeds query, runs cosine similarity search via pgvector, returns ranked note IDs
Frontend: add semantic search input to notes list page, display ranked results with similarity scores
Warning: embedding generation on every note save adds latency. Consider async queue if note volume is high.

Track B — Auto memory compression

Problem it solves: Phase 2 memory compression is manual/triggered — this makes it automatic
New: ai/MemoryCompressionScheduler.java — Spring @Scheduled job, runs nightly at e.g. 2AM
Logic: find all sessions with >20 messages OR last_message_at > 24h ago → for each, call Ollama to summarize → save to ai_chat_summaries → delete raw messages
Add application.properties: ai.compression.enabled=true, ai.compression.cron=0 0 2 * * ?
Add DB index on ai_chat_messages(user_id, created_at) for efficient session lookup
Test on a seeded dataset before enabling in production — summarization quality affects future context quality.

Track C — Long-term memory (goals, habits, preferences)

Problem it solves: AI has no persistent model of the user — can't say "you usually prefer morning tasks" or "your recurring goal is X"
New DB table: ai_user_preferences — user_id, key VARCHAR, value TEXT, updated_at
Keys could include: working_hours, recurring_goals, productivity_patterns, preferred_task_length
New service: ai/UserPreferenceService.java — CRUD on preferences, injected into ContextBuilderService as a new [USER CONTEXT] block
Frontend: optional settings page at /ai/settings for user to view and edit what AI knows about them
Warning: this is the most complex track. Build only if users explicitly ask for more personalised AI behaviour.

Risks Across all phases

Context overflow: hard limit 2000 tokens in ContextBuilderService. Measure, don't estimate.
AI hallucinating IDs: validate every item.id in Ollama response against DB before returning to client. Always.
Ollama cold start: frontend must show loading state. First request after server boot can take 5–15s.
Ollama down: OllamaService must catch all exceptions and return a friendly ChatResponseDto error, not a 500.
AI controlling business logic: AI formats and links only. All filtering, auth, task/note creation = backend only.
Memory creep: full note/task content must never appear in any AI table. Only id + title + due_date in compressed memory.
Overengineering: build P1 completely before writing a single line of P2. Resist the urge to scaffold everything upfront.
Cross-user data leak: every AI service method must scope queries by userId. Never return another user's tasks/notes to AI context.
