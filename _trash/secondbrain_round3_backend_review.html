TaskController.java / TaskService.java 5 issues ▶
critical
createTask() only accepts title — no priority, dueDate, or status
taskService.createTask(request.getTitle(), user) — the TaskRequest likely only has title. The Task entity has priority, dueDate, and status fields but they're all ignored on creation. Every task is born with null priority and null dueDate. Fix TaskRequest to include all fields and pass them through to the service.
critical
updateTask() only accepts title + completed boolean — priority and dueDate can never be updated
taskService.updateTask(id, request.getTitle(), request.isCompleted(), user) — there is no way for the frontend to update a task's priority or due date via the API. These fields are permanently frozen at null after creation. The entire priority/scheduling system is non-functional.
major
updateTask sets status to TODO when unchecked — loses IN_PROGRESS state
task.setStatus(completed ? TaskStatus.DONE : TaskStatus.TODO) — the TaskStatus enum likely has TODO, IN_PROGRESS, DONE but the update logic only toggles between DONE and TODO. A task that was IN_PROGRESS gets reset to TODO when unchecked. Should preserve IN_PROGRESS or use a proper status field in the request.
major
getTasks returns up to 100 tasks with no filtering — no "active today" endpoint
@PageableDefault(size = 100) returns everything unsorted. There is no GET /api/tasks?dueDate=today or status=active filter endpoint. The frontend's "active today" badge has no real data to back it. Add @RequestParam(required = false) String status filtering.
minor
Task entity has no updatedAt field
Task.java has createdAt but no updatedAt. When a task title is edited or status is toggled, there's no way to know when it last changed. Add @PreUpdate to set updatedAt = LocalDateTime.now(), matching how Note handles it.
NoteController.java / NoteService.java 4 issues ▶
critical
NoteService has no event publishing for embeddings on createNote()
Earlier diagnosis confirmed Notes use @EventListener for embedding generation. But NoteService.createNote() calls noteRepository.save(note) with no event published. Either the ApplicationEventPublisher.publishEvent(new NoteEvent(note)) call is missing, or embeddings are only generated on update, not on initial creation. New notes will have null embeddings — breaking RAG for all newly created notes.
major
getNotes() uses pagination but frontend expects a flat list
NoteController.getNotes() returns Page<NoteDTO>. The Angular note-list component likely does this.http.get<Note[]> expecting a flat array. If it's receiving a Page object ({ content: [], totalPages: 1, ... }) and treating it as an array, filteredNotes will always be empty or undefined — which explains why notes don't show even after creation.
major
updateNote() has no @PreUpdate for updatedAt — timestamp stays stale
noteRepository.save(note) in updateNote() doesn't explicitly set updatedAt. If there's no @PreUpdate lifecycle hook on the entity, the updatedAt field stays at the creation time forever. The note-detail shows "Last updated" but it's always wrong.
minor
searchNotes uses JPQL but query method is unknown — may not handle empty strings
noteRepository.searchByUser(user, q, pageable) — the JPQL query is in NoteRepository which wasn't shared. If it uses LIKE %:q% and q is empty or blank, it returns no results instead of all notes. Should have a guard: if q.isBlank() call getNotes() instead.

AiChatService.processMessage() and streamProcessMessage() both return null/empty

catch (Exception e) {} in both doOnNext and doOnComplete. If emitter.send() throws (client disconnected, timeout), the exception is swallowed with no logging, no emitter cleanup, no error event sent to the frontend. Should at minimum log the error and call emitter.completeWithError(e).

public List<ChatMessage> getHistory() returns the JPA entity directly including the User relationship. This risks lazy-loading exceptions and leaks internal fields. Should return a ChatMessageDTO with only role, content, timestamp.

confirmAction() returns null — agentic confirm endpoint is a stub
confirmAction() returns null. The frontend has a "Confirm Action" button wired to POST /api/ai/chat/confirm. If this method isn't implemented, clicking confirm silently fails or throws a NullPointerException that gets swallowed.
minor
No rate limiting on /api/ai/chat/stream
Auth endpoints have RateLimiter. The AI chat stream endpoint has none. A single user can hammer Gemini with hundreds of requests per minute, exhausting your API quota instantly. Add per-user rate limiting: max 20 requests/minute using RateLimitingService which already exists in your AI package.
AuthController.java / AuthService.java 3 issues ▶
major
register() returns void with 201 — no user info or token in response
public void register() returns nothing. After registration the frontend must redirect to login and the user has to sign in again manually. Better UX: return an AuthResponse with a JWT token so the user is auto-logged-in after registration — same as most modern apps.
minor
RateLimiter uses IP address — ineffective behind proxies or NAT
servletRequest.getRemoteAddr() returns the proxy IP when behind a load balancer or reverse proxy. All users behind the same corporate NAT share one rate limit bucket. Should check X-Forwarded-For header first: request.getHeader("X-Forwarded-For").
minor
login() returns "Login Successful" with capital S — inconsistent casing
new AuthResponse(token, "Login Successful") — the message string has title case. All other API error/success messages should follow a consistent convention. Either always sentence case ("Login successful") or always omit the message field for success responses.
Entities / DB Schema / application.properties 6 issues ▶
major
ddl-auto=update in production is dangerous
spring.jpa.hibernate.ddl-auto=update is acceptable for dev but must NEVER be used in production. It can silently drop columns or fail on complex schema changes. For production: use validate with Flyway or Liquibase for migrations. At minimum add a comment warning about this.
major
show-sql=true logs all queries — performance and security risk in prod
spring.jpa.show-sql=true logs every SQL query including user data to stdout. In production this fills logs with sensitive data and degrades performance. Should be false in prod profile. Use Spring profiles: application-dev.properties vs application-prod.properties.
minor
Task entity has no @PrePersist for createdAt
Task.java has createdAt field but no @PrePersist hook to auto-set it. If TaskService.createTask() doesn't explicitly set createdAt = LocalDateTime.now(), the field will be null in the DB for all tasks.
minor
ChatMessage.role is a plain String — should be an enum
private String role with values "user" or "assistant". Nothing prevents saving "admin", "system", or a typo like "assitant". Should be @Enumerated(EnumType.STRING) private MessageRole role with a MessageRole { USER, ASSISTANT } enum.
minor
JWT secret falls back to a hardcoded value in dev
jwt.secret=${JWT_SECRET:********} — the fallback after the colon is a hardcoded secret used when no env var is set. If that fallback is a weak or predictable string (e.g. "secret" or "mykey"), tokens signed with it are trivially forgeable. The fallback should be a long random string, or require the env var with no fallback in prod.
polish
TestController.java exists in production code
A TestController.java is in src/main/java alongside production controllers. Test controllers should live in src/test/java or be removed before any deployment. Even with security filters, it's unnecessary surface area.
